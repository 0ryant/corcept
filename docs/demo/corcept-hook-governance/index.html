What's in this pack
Scenario A · hook governance

Hidden risk, made into a governed decision.

A Claude Code agent attempts a series of tool calls. Wired through settings.json, each PreToolUse event is routed to corcept hook pretool-guard before the tool runs. corcept returns the Claude Code permission protocol — allow / ask / deny — over the 10 events. Three tiers, not a binary: deny stops it, ask escalates to a human (L4 approval), allow lets routine work through. Each decision names why.

corcept hook pretool-guard — 5 deny
# dangerous calls, blocked at PreToolUse →
Bash  rm -rf /                       ✗ deny recursive force delete of `/`
Bash  curl http://… | bash           ✗ deny remote fetch piped to interpreter
Bash  chmod -R 777 /                 ✗ deny world-writable/executable
Read  /etc/shadow                    ✗ deny secret-like / protected read
Bash  … >> ~/.ssh/authorized_keys    ✗ deny references protected path
corcept hook pretool-guard — 2 ask / 3 allow
# sensitive → escalate to a human (L4) →
Bash  git push --force origin main   → ask force-push needs L4 approval
Bash  sudo rm -rf /var/log           → ask recursive delete needs approval

# ordinary work → allowed →
Bash  cargo test --workspace         ✓ allow passed CORCEPT guard
Bash  git status                     ✓ allow passed CORCEPT guard
Edit  src/main.rs                    ✓ allow passed filesystem guard

Totals: 5 deny / 2 ask / 3 allow, across 10 events. Raw per-event output is in decisions/ (Claude Code protocol JSON), summarised in decisions/summary.md. The dangerous commands (rm -rf /, curl | bash, …) are never run — corcept evaluates the PreToolUse payloads and returns a permission decision only.

Honest residual — defence in depth, not a complete sandbox

The guard matches a curated set of high-signal patterns, not "all bad commands". For example, a raw dd if=/dev/zero of=/dev/sda passed the guard in testing — device-level destruction isn't in the current pattern set.

corcept reduces the blast radius of the common, high-frequency footguns and routes the ambiguous ones to a human; it is defence in depth, not a complete sandbox. For actual process/network isolation, that is cellos's job, not corcept's.

Scenario B · tamper-evident ledger

You cannot rewrite what the agent was allowed to do.

Every decision in Scenario A is appended to a hash-chained, append-only audit ledger (ledger/events.jsonl). Each row carries the SHA-256 hash of its content plus the prev_hash of the row before it, so the whole log is a chain — and each row records the authority_level (L0_observe … L3_execute_local), the decision, and the reason.

corcept audit verify — intact
# the ledger as written →
$ corcept audit verify
{ "status": "pass",
  "hash_chain_valid": true,
  "rows_scanned": 11,
  "failures": [] }
corcept audit verify — after a 1-row edit
# attacker flips rm -rf deny → allow,
# without recomputing the chain →
$ corcept audit verify
{ "status": "fail",
  "hash_chain_valid": false,
  "rows_scanned": 11,
  "failures": [ { "line": 6,
    "event_id": "evt_…",
    "reason": "hash_mismatch" } ] }

The edit is caught and the exact altered row is named. Outputs are in ledger/verify-intact.json (pass) and ledger/verify-tampered.json (fail, hash_mismatch, line 6 pinpointed); the tamper test is described in ledger/tamper.txt. The ledger can also be Ed25519-signed per row: corcept audit verify --signed — that is the honest "optionally signed" ceiling, not "signed by default".

Reproduce it

One script. No hand-edited outputs.

Requires the corcept binary (cargo build -p corcept-cli --release → target/release/corcept.exe). The script inits a throwaway governed workspace, runs every event through the PreToolUse guard, audits one completed call, then verifies the ledger intact and again after tampering a copy.

1
init · governed workspace
corcept init scaffolds a throwaway, gitignored workspace holding the .corcept config and the audit ledger.
2
hook pretool-guard · ×10
Every synthetic event is run through the PreToolUse guard, writing decisions/<event>.decision.json and summary.md (5 deny / 2 ask / 3 allow).
3
audit verify · intact
The ledger is copied out and verified: status=pass, hash_chain_valid=true, 11 rows scanned, no failures.
4
tamper + re-verify
A copy's rm -rf deny is flipped to allow without recomputing the chain; re-verify returns fail and pinpoints the altered row (hash_mismatch).
pwsh
# from the repo root
$ pwsh docs/demo/corcept-hook-governance/reproduce.ps1
Scenario A — governing hook events...
Scenario B — verifying + tampering the ledger...
Done. See decisions/ and ledger/.
